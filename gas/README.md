# GAS移行 - 写真貼付＋Excel出力 検証手順

## このディレクトリの位置づけ
GAS移行で最大の技術的リスクである「写真をセル範囲にぴったり貼り付けてxlsxとして
書き出す」処理が、GAS単体で実現可能かどうかを検証するためのスクリプト。
ここがクリアできれば、運転記録表本体（template.xlsx・写真なし）の方は
セル書き込みだけなので難易度は低い。

## セットアップ手順

1. **新しいGoogle Apps Scriptプロジェクトを作成**
   https://script.google.com → 新しいプロジェクト

2. **このフォルダの2ファイルを貼り付け**
   - `appsscript.json`（プロジェクトの設定 →「マニフェスト ファイルを表示」を
     有効化してから内容を置き換える）
   - `PhotoExportTest.gs`（エディタにそのままコピー）

3. **photo_template.xlsx をGoogleスプレッドシート形式に変換**
   - Googleドライブに `photo_template.xlsx` をアップロード
   - 右クリック →「アプリで開く」→「Google スプレッドシート」
   - 変換された新しいスプレッドシートのURLからIDをコピー
     （`https://docs.google.com/spreadsheets/d/【このID】/edit`）

4. **出力先のGoogle Driveフォルダを用意し、そのIDもコピー**

5. **`setupScriptProperties` 関数内のIDを書き換えて1回実行**
   - 実行後、関数内の値は消してコミットしないこと（IDは個人のDrive環境に依存するため）

6. **`testPhotoExport` を実行**
   - 初回実行時は権限の許可画面が出るので許可する
   - 実行ログ（表示 →「実行ログ」）にxlsxファイルのURLが出力される

7. **出力されたxlsxファイルをダウンロードし、Excelで開いて確認**
   - 3枚のサンプル画像がB〜F列×指定行範囲にぴったり収まっているか
   - 日付・キャプションがH列の正しい行に入っているか

## 確認したいポイント

- [ ] 画像がセル範囲の境界からズレていないか（GAS版とExcel版で見た目が一致するか）
- [ ] xlsxエクスポート後も画像がセルに対して相対位置を保っているか
   （Googleスプレッドシート上の「絶対位置」画像が、xlsx変換時にセル基準の
   アンカーに変換されるかどうかが未検証ポイント）
- [ ] 16シート×3枚＝48枚規模でも処理時間・タイムアウト（GASは1回の実行が
   6分まで）に収まるか

## 次のステップ
この検証がOKであれば、本体の運転記録表（template.xlsx相当）のセル書き込みと
合体させ、スプレッドシートDBから1レコード分のデータを読んで出力する
本実装に進む。

## 運転記録表本体（OperationRecordExportTest.gs）の検証

写真台帳とは別に、運転記録表本体（template.xlsx、3シート構成: 1-4号機/
5-8号機/9-12号機）へのセル書き込みを検証するスクリプト。
`PhotoExportTest.gs`と同じプロジェクト内に置いて使う
（`exportSpreadsheetAsXlsx`関数を共用するため）。

### セットアップ

1. `template.xlsx` をGoogleドライブにアップロード→「アプリで開く」→
   「Google スプレッドシート」で変換（3シート構成がそのまま保持される）
2. 変換後スプレッドシートのIDをコピー
3. `OperationRecordExportTest.gs` を同じApps Scriptプロジェクトに追加
   （ファイル追加: 左の「ファイル」横の「+」→「スクリプト」）
4. `setupOperationRecordProperties` 内のIDを書き換えて1回実行
   （`OUTPUT_FOLDER_ID`は`PhotoExportTest.gs`側で設定済みのものを再利用するので、
   この関数では`OPERATION_TEMPLATE_SHEET_ID`のみ設定すればよい）
5. `testOperationRecordExport` を実行
6. 出力されたxlsxを開き、サンプルデータ（現場名・住所・1号機のデータなど）が
   正しいセルに入っているか確認する

### 確認したいポイント

- [ ] 基本情報（現場名D2、住所D3など）が正しいセルに入っているか
- [ ] 1号機の各回路（A〜D）のデータがG〜J列に正しく分散しているか
- [ ] チェック項目（■/□）が選択状態を正しく反映しているか
- [ ] マージセルになっている箇所（D2など結合されている可能性のあるセル）で
      エラーが出ないか、値が正しい位置に入るか

## DB（記録保存・一覧）の検証（Database.gs / DatabaseTest.gs）

現行PostgreSQL版の `records` / `photos` テーブルを、スプレッドシートDBで
再現したもの。写真本体はスプレッドシートのセルに直接入れず、Google Drive
専用フォルダにファイルとして保存し、シート側にはDriveファイルIDのみ持たせる。

### セットアップ

1. **新しい空のGoogleスプレッドシートを1つ作成**（DB専用、名前は任意。例: `USX_DB`）
   そのスプレッドシートのIDをコピー
2. **写真保存用のGoogle Driveフォルダを1つ新規作成**
   （PhotoExportTest.gsのOUTPUT_FOLDER_ID＝検証用の出力先とは別物。
   こちらは本番データを永続的に保存する場所なので、混同しないよう注意）
   そのフォルダのIDをコピー
3. 同じApps Scriptプロジェクトに `Database.gs` と `DatabaseTest.gs` を追加
4. `Database.gs` 内の `setupDatabaseProperties` のIDを書き換えて実行
5. `initDb` を実行（records / photos シートが作成される。実行ログのURLを開いて
   2つのシートができていることを確認）
6. `testDatabaseCrud` を実行し、実行ログを上から順に確認

### 確認したいポイント

- [ ] 記録の新規保存→一覧取得→1件取得→更新→削除が全てエラーなく流れるか
- [ ] 写真の保存→一覧取得→本体取得→削除が正しく動くか
- [ ] 削除した記録・写真が、Drive上のファイルも含めてきちんと消えているか
- [ ] DBスプレッドシートを開いて、records/photosシートの中身が想定通りか目視確認

### 注意点（運用前に判断したいこと）

- 複数人が同時に保存した場合の競合（同時書き込み）はまだ未検証。
  必要であれば `LockService` での排他制御を後から追加する
- `getRecords()` は毎回シート全体を読み込む実装。記録件数が増えてきたら
  パフォーマンスを見て調整する
